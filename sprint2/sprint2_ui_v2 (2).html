Edge Cases & Failure Handling
EC-01: Duplicate Email
Attempts to register a user with an email that already exists in the system.
INSERT INTO users (email, password_hash, role) VALUES ('alice.johnson@email.com', 'pw', 'user');
EC-02: Invalid Booking Time
Attempts to create a booking where end_time is before start_time.
INSERT INTO bookings (user_id, slot_id, start_time, end_time, status) VALUES (1, 1, '2025-05-01 10:00', '2025-05-01 08:00', 'reserved');
EC-03: Duplicate Payment
Attempts to insert a second payment for booking #1, which already has a payment.
INSERT INTO payments (booking_id, amount_cents, currency, status, provider) VALUES (1, 500, 'USD', 'pending', 'Stripe');
EC-04: Invalid Role
Attempts to register a user with a role value outside 'user' or 'admin'.
INSERT INTO users (email, password_hash, role) VALUES ('hacker@email.com', 'pw', 'superadmin');
EC-05: Delete User With Bookings
Attempts to hard-delete a user who has existing booking records.
DELETE FROM users WHERE user_id = 1; -- FK on bookings.user_id blocks this
EC-06: Invalid Slot Status
Attempts to set a parking slot status to a value not in the allowed list.
UPDATE parking_slots SET status = 'broken' WHERE slot_id = 1; -- CHECK constraint rejects 'broken'
EC-07: Dispute with Non-Existent Booking
Attempts to open a dispute referencing a booking that does not exist.
INSERT INTO disputes (booking_id, opened_by_user_id, reason, status) VALUES (9999, 1, 'test', 'open'); -- FK on disputes.booking_id rejects this
EC-08: Soft Delete (Correct Approach)
Shows the correct way to deactivate a user without violating FK constraints.
-- WRONG: DELETE FROM users WHERE user_id = 1; -- CORRECT soft delete: UPDATE users SET is_active = 0, updated_at = SYSDATE WHERE user_id = 1;

AQ1 — Full Booking Report (Users + Slots JOIN)

Scenario: Admin views all bookings with user names and slot details.

SELECT b.booking_id, u.first_name || ' ' || u.last_name AS user_name, ps.slot_code, ps.level_name, ps.zone, b.start_time, b.end_time, b.status FROM bookings b JOIN users u ON b.user_id = u.user_id JOIN parking_slots ps ON b.slot_id = ps.slot_id ORDER BY b.start_time DESC;

Why JOIN: booking records store only IDs; names and slot codes live in separate tables.

AQ2 — Transaction Report (4-table JOIN)

Scenario: Finance team views bookings with vehicle and payment details.

SELECT b.booking_id, u.email, v.plate_number, v.make || ' ' || v.model AS vehicle, p.amount_cents / 100 AS amount_usd, p.status AS payment_status, p.paid_at FROM bookings b JOIN users u ON b.user_id = u.user_id JOIN vehicles v ON b.vehicle_id = v.vehicle_id JOIN payments p ON b.booking_id = p.booking_id ORDER BY p.paid_at DESC NULLS LAST;

Why JOIN: requires 4 tables — users, vehicles, bookings, payments — to produce one transaction row.

AQ3 — Available Slots for a Time Window

Scenario: User searches for open slots on May 1st 10am–12pm.

SELECT ps.slot_id, ps.slot_code, ps.level_name, ps.zone FROM parking_slots ps WHERE ps.status = 'active' AND ps.slot_id NOT IN ( SELECT b.slot_id FROM bookings b WHERE b.status NOT IN ('cancelled','completed','no_show') AND b.start_time < TO_DATE('2025-05-01 12:00','YYYY-MM-DD HH24:MI') AND b.end_time > TO_DATE('2025-05-01 10:00','YYYY-MM-DD HH24:MI')) ORDER BY ps.slot_code;

Why JOIN: subquery correlates parking_slots with bookings to detect overlapping reservations.

AQ4 — Revenue Per User (GROUP BY)

Scenario: Business analyst identifies highest-spending users.

SELECT u.user_id, u.first_name || ' ' || u.last_name AS user_name, COUNT(p.payment_id) AS total_bookings, SUM(p.amount_cents) / 100 AS total_spent_usd FROM users u JOIN bookings b ON u.user_id = b.user_id JOIN payments p ON b.booking_id = p.booking_id WHERE p.status = 'paid' GROUP BY u.user_id, u.first_name, u.last_name ORDER BY total_spent_usd DESC;

Why GROUP BY: aggregates payment totals per user across multiple bookings.

AQ5 — Dispute Dashboard (LEFT JOIN)

Scenario: Admin sees all disputes with opener, assigned admin, and slot.

SELECT d.dispute_id, op.first_name || ' ' || op.last_name AS opened_by, ad.first_name || ' ' || ad.last_name AS assigned_admin, ps.slot_code, b.start_time, d.reason, d.status FROM disputes d JOIN bookings b ON d.booking_id = b.booking_id JOIN users op ON d.opened_by_user_id = op.user_id LEFT JOIN users ad ON d.assigned_admin_id = ad.user_id JOIN parking_slots ps ON b.slot_id = ps.slot_id ORDER BY d.created_at DESC;

Why LEFT JOIN: admin may be NULL for open disputes — LEFT JOIN prevents those rows from being excluded.

AQ7 — Users With Open Disputes (GROUP BY)

Scenario: Trust team flags accounts with unresolved disputes.

SELECT u.user_id, u.first_name || ' ' || u.last_name AS user_name, u.email, COUNT(d.dispute_id) AS open_disputes FROM users u JOIN bookings b ON u.user_id = b.user_id JOIN disputes d ON b.booking_id = d.booking_id WHERE d.status IN ('open','in_review') GROUP BY u.user_id, u.first_name, u.last_name, u.email ORDER BY open_disputes DESC;

Why 2 JOINs: disputes link to bookings, not directly to users — two JOINs needed to trace back to the user.

AQ10 — Daily Bookings by Zone (GROUP BY)

Scenario: Operations team analyzes zone utilization by day.

SELECT ps.zone, TRUNC(b.start_time) AS booking_date, COUNT(b.booking_id) AS bookings_that_day FROM bookings b JOIN parking_slots ps ON b.slot_id = ps.slot_id WHERE b.status NOT IN ('cancelled','no_show') GROUP BY ps.zone, TRUNC(b.start_time) ORDER BY booking_date, ps.zone;

Why GROUP BY: aggregates booking counts per zone per day across two tables.
